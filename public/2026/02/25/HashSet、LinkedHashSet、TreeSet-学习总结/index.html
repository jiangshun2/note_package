HashSet、LinkedHashSet、TreeSet - 学习总结

HashSet、LinkedHashSet、TreeSet - 学习总结

学习日期：2026-02-25
学习时长：约 15 分钟
学习方法：费曼学习法（问答互动）

一、今日学习内容汇总

1. HashSet

知识点：

  • 底层实现：哈希表（数组 + 链表/红黑树）
  • 特点：
    • ❌ 无序（不保证元素的存储顺序）
    • ❌ 不可重复（保证元素唯一性）
    • ❌ 无索引（Set 接口都没有索引）
    • ✅ 允许一个 null 值
  • 性能：添加、删除、查询都是 O(1)，性能最快

去重机制（重要！）：

关键方法：

  • 必须同时重写 hashCode() 和 equals() 方法
  • 只重写 equals()：两个对象相等但哈希值不同，会被存储在不同位置，导致重复
  • 只重写 hashCode()：两个对象哈希值相同但不相等，会被误判为重复

记忆口诀：

HashSet 哈希表，无序不可重复
去重靠哈希，equals 定乾坤

实际应用场景：

  • ✅ 只需要去重，不关心顺序
  • ✅ 对性能要求极高
  • ✅ 用户标签系统、权限集合、数据去重

我的理解：
HashSet 是最常用的 Set 实现，底层使用哈希表，所以添加、删除、查询都非常快，时间复杂度是 O(1)。但是元素是无序的，不能保证存储顺序。HashSet 通过 hashCode() 和 equals() 方法来判断元素是否重复，所以如果要让自定义类的对象能够正确去重，必须同时重写这两个方法。HashSet 适合只需要去重而不关心顺序的场景，比如用户标签系统，一个标签不能重复添加。

2. LinkedHashSet

知识点：

  • 底层实现：哈希表 + 双向链表
  • 特点：
    • ✅ 有序（按插入顺序）
    • ❌ 不可重复（保证元素唯一性）
    • ❌ 无索引（Set 接口都没有索引）
    • ✅ 允许一个 null 值
  • 性能：添加、删除、查询都是 O(1)，略慢于 HashSet（因为需要维护链表）

与 HashSet 的区别：

  • LinkedHashSet 比 HashSet 多了一个双向链表，用于维护插入顺序
  • 在添加元素时，除了存储到哈希表，还会在链表末尾添加节点
  • 在遍历时，按照链表的顺序（即插入顺序）遍历

记忆口诀：

LinkedHashSet 插序，哈希链表双保险
有序不重复，历史记录用

实际应用场景：

  • ✅ 需要去重，且关心插入顺序
  • ✅ 历史记录、最近访问列表
  • ✅ LRU 缓存的实现
  • ✅ 购物车商品列表（去重且保持添加顺序）

我的理解：
LinkedHashSet 在 HashSet 的基础上增加了一个双向链表来维护插入顺序，所以元素是按照插入顺序排列的。这使得 LinkedHashSet 既能去重，又能保持插入顺序，非常适合用于需要记录历史记录的场景，比如浏览器的浏览历史、最近访问的商品列表等。虽然性能略低于 HashSet，但在需要保持顺序的场景下是非常好的选择。

3. TreeSet

知识点：

  • 底层实现：红黑树（注意：不是哈希表！）
  • 特点：
    • ✅ 有序（按自然顺序或自定义排序）
    • ❌ 不可重复（保证元素唯一性）
    • ❌ 无索引（Set 接口都没有索引）
    • ❌ 不允许 null 值
  • 性能：添加、删除、查询都是 O(log n)，性能较慢

去重和排序机制（重要！）：

关键方法：

  • TreeSet 不使用 hashCode() 和 equals() 方法
  • 使用 compareTo() 方法（自然排序）或 compare() 方法（比较器排序）

排序方式：

方式 1：自然排序（Comparable）

  • 让元素类实现 Comparable 接口
  • 重写 compareTo() 方法
  • 适用于元素类有固定的排序规则

方式 2：比较器排序（Comparator）

  • 创建 Comparator 对象（匿名内部类或 Lambda）
  • 在创建 TreeSet 时传入比较器
  • 适用于需要临时改变排序规则，或无法修改元素类
  • 优先级高于自然排序

记忆口诀：

TreeSet 红黑树，有序不重复
排序两方式，Comparable 比较器
性能 O(log n)，排行榜上用

实际应用场景：

  • ✅ 需要排序的集合
  • ✅ 排行榜、成绩排名、商品价格排序
  • ✅ 需要快速查找范围（如查找某个区间内的元素）
  • ❌ 只关心去重，不关心顺序（应该用 HashSet）
  • ❌ 需要插入顺序（应该用 LinkedHashSet）

我的理解：
TreeSet 底层使用红黑树，元素会按照自然顺序或自定义排序规则排列，所以是有序的。TreeSet 通过 compareTo() 或 compare() 方法来判断元素是否重复和确定排序位置，而不是使用 hashCode() 和 equals() 方法。TreeSet 适合需要排序的场景，比如排行榜、成绩排名等。但是 TreeSet 的性能不如 HashSet 和 LinkedHashSet，因为红黑树的插入、删除、查询时间复杂度是 O(log n)，而哈希表是 O(1)。

二、三种 Set 的对比

特性 HashSet LinkedHashSet TreeSet
底层结构 哈希表（数组+链表/红黑树） 哈希表 + 双向链表 红黑树
是否有序 ❌ 无序 ✅ 有序（按插入顺序） ✅ 有序（按排序规则）
是否允许 null ✅ 允许一个 null ✅ 允许一个 null ❌ 不允许 null
是否允许重复 ❌ 不允许 ❌ 不允许 ❌ 不允许
是否有索引 ❌ 无索引 ❌ 无索引 ❌ 无索引
添加/删除/查询 ⚡ O(1) 最快 ⚡ O(1) 较快 🐢 O(log n) 较慢
遍历 ⚡ O(n) ⚡ O(n) ⚡ O(n)

六、学习心得

通过今天的学习，我对 HashSet、LinkedHashSet 和 TreeSet 三个 Set 接口的实现类有了更深入的理解。以前只知道它们都是 Set 的实现，都能去重，现在清楚了它们的底层实现原理、特点和适用场景。

HashSet 是最常用的 Set 实现，底层使用哈希表，性能最好，但是无序。LinkedHashSet 在 HashSet 的基础上增加了一个双向链表来维护插入顺序，所以是有序的，但性能略低于 HashSet。TreeSet 底层使用红黑树，元素会按照自然顺序或自定义排序规则排列，所以是有序的，但性能不如前两者。

特别重要的是，HashSet 和 LinkedHashSet 使用 hashCode() 和 equals() 方法来判断元素是否重复，而 TreeSet 使用 compareTo() 或 compare() 方法。这个区别在面试中经常被问到，也是实际开发中需要注意的地方。
