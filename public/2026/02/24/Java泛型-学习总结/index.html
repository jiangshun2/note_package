一、今日学习内容汇总

1. 泛型的概念和作用

知识点：

  • 泛型是 Java 5 引入的特性，使用 <类型参数> 语法
  • 核心作用：
    • 编译时类型检查，避免运行时 ClassCastException
    • 避免类型转换（不需要手动 cast）
    • 增强代码的可读性和复用性

记忆口诀：

泛型就是尖括号，类型参数往里填
编译时检查类型，运行时无转换

实际应用场景：

  • ✅ 集合框架：List<String>、Map<String, User>
  • ✅ 工具类：Optional<T>、Function<T, R>
  • ✅ 统一响应体：Response<T>

我的理解：
泛型就像一个类型占位符，在定义类、接口或方法时，用一个符号（如 T、E）来表示某个类型，在使用时再指定具体的类型。这样做的好处是可以在编译时就检查类型是否正确，而不是等到运行时才发现错误。比如，当我们使用 List<String> 时，编译器会检查我们添加的元素是否都是字符串类型，如果不是就会报错，这样就避免了运行时的类型转换错误。

3. 泛型的通配符

知识点：

通配符 名称 作用 适用场景
? 无界通配符 代表任意类型 只读取，不写入
? extends T 上界通配符 代表 T 或 T 的子类 只能读，不能写（只读）
? super T 下界通配符 代表 T 或 T 的父类 只能写，不能读（只写）

记忆口诀：

上界 extends 只读，下界 super 只写
无界通配符 ?，任意类型都可以

实际应用场景：

  • ✅ ? extends T：读取集合中的元素（如 printList(List<? extends Number>)）
  • ✅ ? super T：向集合中添加元素（如 addToList(List<? super String>, String)）
  • ✅ ?：传递集合，不关心具体类型（如 processList(List<?>)）

我的理解：
通配符是用来表示未知类型的，? 表示任意类型，? extends T 表示 T 或 T 的子类，? super T 表示 T 或 T 的父类。上界通配符适用于只读取的场景，因为我们不知道具体的子类类型，所以不能写入；下界通配符适用于只写入的场景，因为我们不知道具体的父类类型，所以不能读取。

4. 类型擦除

知识点：

  • 类型擦除：编译时，泛型信息被擦除，生成与非泛型代码类似的字节码
  • 影响：
    • 运行时无法获取泛型类型信息
    • 编译器会在必要时自动插入类型转换代码
    • 可能导致类型安全问题（如使用反射绕过检查）

记忆口诀：

泛型尖括号类型定，编译检查运行擦
运行时无泛型信息，类型转换隐式执行

实际应用场景：

  • ✅ 了解类型擦除有助于理解泛型的内部实现
  • ✅ 避免在运行时尝试获取泛型类型信息

我的理解：
类型擦除是 Java 泛型的实现方式，它保证了与旧代码的兼容性。编译时，编译器会检查泛型类型是否正确，然后擦除泛型信息，生成与非泛型代码类似的字节码。运行时，JVM 看不到泛型信息，只看到原始类型。这就是为什么我们不能在运行时使用 instanceof 检查泛型类型，也不能创建泛型数组。

5. 泛型的限制

知识点：

  • ✅ 只能是引用类型：基本类型需要用包装类（如 int → Integer）
  • 不能创建泛型数组：如 new T[5] 会编译错误
  • 不能在静态上下文中使用泛型类型参数：静态字段和方法不能使用类的泛型参数
  • 不能使用 instanceof 检查泛型类型：如 if (obj instanceof List<String>) 会编译错误

记忆口诀：

基本类型用包装，静态上下文中无
泛型数组不能建，instanceof 检查无

实际应用场景：

  • ✅ 使用包装类处理基本类型（如 List<Integer> 而不是 List<int>）
  • ✅ 使用 List<T> 而不是泛型数组

我的理解：
泛型有一些限制，主要是由于类型擦除导致的。例如，泛型只能是引用类型，因为基本类型在运行时没有类型信息，无法进行类型擦除。泛型数组不能创建，因为数组是协变的，而泛型是不变的，可能会导致类型安全问题。静态上下文中不能使用泛型类型参数，因为静态成员属于类，而不是实例，而泛型类型参数是在实例化时才确定的。

6. 泛型与继承的关系

知识点：

  • 泛型是不变的：List<String> 不是 List<Object> 的子类型
  • 原因：类型擦除后，List<String> 和 List<Object> 都是 List，如果允许它们有继承关系，可能会导致类型安全问题
  • 正确做法：使用通配符

记忆口诀：

泛型不变要记牢，类型擦除是原因
需要转换用通配，安全又可靠

实际应用场景：

  • ✅ 当需要处理不同类型的泛型集合时，使用通配符而不是类型转换

我的理解：
泛型是不变的，这意味着即使 String 是 Object 的子类，List<String> 也不是 List<Object> 的子类。这是因为类型擦除后，它们都是 List 类型，如果允许它们有继承关系，可能会导致类型安全问题。例如，如果我们可以将 List<String> 赋值给 List<Object>，然后向其中添加一个 Integer，那么在取出元素时就会出现类型转换错误。正确的做法是使用通配符，如 List<?> 或 List<? extends Object>。

7. 泛型的桥接方法

知识点：

  • 桥接方法：编译器为解决泛型类型擦除导致的“重写失效”问题，自动生成的隐式方法
  • 产生原因：泛型类型擦除后，子类的泛型重写方法签名与父类不匹配，编译器生成桥接方法补全多态逻辑

示例：

记忆口诀：

桥接方法编译器生，类型擦除重写保
多态逻辑补全后，运行时行为正确

实际应用场景：

  • ✅ 桥接方法是编译器自动处理的，开发者无需关心
  • ✅ 了解桥接方法有助于理解泛型的内部实现

我的理解：
桥接方法是编译器为了解决泛型类型擦除导致的重写失效问题而自动生成的。当我们在子类中重写父类的泛型方法时，由于类型擦除，子类的方法签名与父类的方法签名不匹配，这样就会导致多态失效。编译器为了解决这个问题，会自动生成一个桥接方法，该方法的签名与父类的方法签名匹配，然后在桥接方法中调用子类的重写方法。这样，运行时的多态行为就可以正常工作了。

8. 泛型的典型应用场景

知识点：

  1. 集合框架：

    • List<String>、Map<String, User>、Set<Integer> 等
    • 避免类型转换，提高类型安全

  2. 工具类：

    • Collections.sort(List<T>)：排序任意类型的列表
    • Arrays.asList(T...)：将数组转换为列表
    • Optional<T>：处理可能为 null 的值

  3. 统一响应体/结果封装：

  4. 函数式接口：

    • Function<T, R>：转换类型
    • Predicate<T>：判断条件
    • Consumer<T>：消费数据

  5. 数据访问层：

    • Dao<T>：通用数据访问接口，如 UserDao extends Dao<User>

记忆口诀：

集合工具响应体，函数接口数据层
泛型应用无处不在，类型安全又简洁

实际应用场景：

  • ✅ 几乎所有现代 Java 代码都在使用泛型
  • ✅ 提高代码的可读性、可维护性和类型安全性

我的理解：
泛型在 Java 中的应用非常广泛，几乎所有现代 Java 代码都在使用泛型。它可以帮助我们编写更简洁、更安全、更可复用的代码。例如，在集合框架中，我们可以使用 List<String> 来表示一个字符串列表，这样编译器就会检查我们添加的元素是否都是字符串类型，避免了运行时的类型转换错误。在工具类中，我们可以使用泛型方法来处理任意类型的数据，提高代码的复用性。在统一响应体中，我们可以使用泛型来封装不同类型的响应数据，使代码更简洁。
