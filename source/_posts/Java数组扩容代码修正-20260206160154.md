✅ System.exit(int status) 的作用 立即终止当前运行的`Java`虚拟机（JVM）。 在退出前，会执行： 所有已注册的 shutdown hooks（通过 Runtime.getRuntime().addShutdownHook() 注册）； 执行 finalizers（如果启用了 finalization）； 关闭系统资源（如标准输入输出流等）。

⚠️ 一旦调用 System.exit()，程序立刻开始终止流程，不会等当前方法或线程自然结束！

✅ 状态码（status）的含义 状态码 | 含义 ——–|——**0**| 表示 正常退出（success）。这是约定俗成的标准。 非 0（如 1, -1, 99 等） | 表示 异常退出 / 错误退出（failure）。具体数值可自定义，用于告诉操作系统或父进程“出错了”。 📌 例如在 Linux/Shell 中： >`java`MyApp echo $? # 查看上一个命令的退出状态码 如果程序调用 System.exit(5)，那么 echo $? 会输出 5。

→ 你不应该替调用者决定是否退出 JVM。应抛出异常，让上层处理。 在多线程环境中，除非你确定要全局退出

✅ 最佳实践：

## assistant

这是一个非常核心且实用的问题！在`Java`中，像 ArrayList、StringBuilder 等动态结构都依赖数组扩容，而扩容本身代价很高（需要分配新内存 + 拷贝所有元素），所以必须通过策略优化来减少性能损耗。

下面我们从 原理 + 实践 两个层面，系统讲解数组扩容的性能优化方法：

🔸 1. 避免频繁扩容 → 采用“倍增策略”（Geometric Growth）

❌ 错误做法：每次只扩 1 个 // 千万别这样！ int[] newData = new int[capacity + 1]; 插入第 n 个元素时，要拷贝 n-1 个元素； 插入 N 个元素总拷贝次数 ≈ 1+2+3+…+N = O(N²) → 性能灾难！

✅ 正确做法：按比例扩容（通常是 1.5 倍 或 2 倍） int newCapacity = capacity + (capacity >> 1); // 1.5 倍（JDK ArrayList 默认） // 或 int newCapacity = capacity * 2; // 2 倍（简单高效）

优点： 扩容次数大大减少：插入 N 个元素，只需扩容 O(log N) 次； 总拷贝次数 ≈ N + N/2 + N/4 + … < 2N → 摊还时间复杂度 O(1)！

📌 JDK 的 ArrayList 在`Java`8+ 中使用 1.5 倍扩容（更省内存），而早期版本或某些场景用 2 倍。

🔸 2. 预设初始容量（Initial Capacity）

如果你大概知道数据量，创建时就指定足够大的初始容量：

// 坏：默认容量 10，插入 10000 个元素会多次扩容 List`Integer` list = new ArrayList<>();

// 好：直接给够 List`Integer` list = new ArrayList<>(10000);

✅ 效果：完全避免扩容，性能最优！

💡 经验：在循环前收集数据量，或根据业务估算（如分页大小 × 页数）。

🔸 3. 使用 System.arraycopy() 而非 for 循环

你已经知道这一点了 👍
System.arraycopy() 是 native 方法，底层用 C/C++ 实现，可能使用： 内存块复制（memcpy） SIMD 指令（一次拷贝多个字节） 页对齐优化

比`Java`的 for 循环快 5~10 倍以上，尤其在大数据量时。

🔸 4. 避免不必要的扩容（比如先检查是否真的满）

在 insert 或 add 前，只在必要时才扩容： if (size >= capacity) { grow(); // 扩容 } 而不是每次都创建新数组。

🔸 5. 考虑使用更高效的数据结构（如果频繁增删）

如果频繁在中间插入/删除，数组即使扩容快，移动元素也慢（O(n)）。

可考虑： LinkedList（但随机访问慢） 分段数组（如 ArrayDeque 的环形缓冲区） 第三方库（如 Eclipse Collections、FastUtil）

⚠️ 但注意：90% 的场景 ArrayList + 合理初始容量 已足够高效。

🔸 6. JVM 层面优化（你无法控制，但要知道） TLAB（Thread Local Allocation Buffer）：JVM 为每个线程预分配一块堆内存，使得 new int[N] 几乎无锁、极快； 逃逸分析 + 栈上分配：如果数组不会逃逸出方法，JVM 可能把它分配在栈上，避免 GC 压力。

✅ 总结：数组扩容性能优化 checklist 优化手段 | 是否推荐 | 说明 ——–|——–|—— ✅ 使用倍增策略（1.5x 或 2x） | ✔️ 必须 | 摊还 O(1) ✅ 预设合理初始容量 | ✔️ 强烈推荐 | 避免扩容 ✅ 用 System.arraycopy() | ✔️ 必须 | 底层高效 ✅ 仅在必要时扩容 | ✔️ 基础 | 避免浪费 ⚠️ 考虑替代数据结构 | 视场景 | 非通用方案 ❌ 每次只扩 1 个 | ✖️ 禁止 | O(N²) 灾难 🎯 举个实际例子（来自 JDK ArrayList.grow()）：

这就是工业级的扩容优化！

📌 关键概念区分： 变量 | 含义 | 合法下标范围 ——|——|————- size | 已存储的元素个数 | 下一个空位是 size（只要 size < capacity） capacity | 数组总长度（最大容量） | 最大下标是 capacity - 1 ✅ 所以：size 既是“元素个数”，也是“下一个插入位置的索引”。

这其实是数组实现栈或动态列表的经典技巧！

💡 类比理解： 想象一个水杯： capacity = 杯子总高度（比如 10cm） size = 当前水位（比如 3cm） 下一滴水应该加在 3cm 的位置，然后水位变成 4cm

data[size] = value; size++; 就是这个过程！

✅ 总结一句话：

因为数组下标从 0 开始，存了 size 个元素，就占用了 0 到 size-1，所以下一个空位置就是 size。
